```dart
// Pure helper mirroring the payload construction inside
// AuditTemplateService.reorderItems. Kept in sync manually — the

List<Map<String, dynamic>> buildReorderPayload(List<String> ids) {
```

```dart
  /// Reordena itens atualizando [order_index] de cada um sequencialmente.
  /// Recebe a lista de IDs na nova ordem desejada.
  Future<void> reorderItems(List<String> ids) async {
    for (int i = 0; i < ids.length; i++) {
      await _client
          .from('template_items')
          .update({'order_index': i})
          .eq('id', ids[i]);
    }
  }
```

Replacement:
```dart
  /// Reordena itens atualizando [order_index] via batch upsert em 1 query (PERF-01).
  /// Recebe a lista de IDs na nova ordem desejada.
  /// Todos os IDs devem existir em `template_items` — IDs inválidos causam
  /// erro de constraint (não silencioso).
  Future<void> reorderItems(List<String> ids) async {
    if (ids.isEmpty) return;
    final payload = [
      for (int i = 0; i < ids.length; i++)
        {'id': ids[i], 'order_index': i},
    ];
    await _client.from('template_items').upsert(payload);
  }
```

Notes:
- No `onConflict` parameter: PostgREST defaults to the table's PRIMARY KEY (`id`), which is exactly what we want. Adding `onConflict: 'id'` would be redundant — do NOT add it.
- No try/catch: CLAUDE.md convention ("services do not handle exceptions internally"). Callers handle errors.
- The guard clause `if (ids.isEmpty) return;` mirrors defensive patterns used elsewhere in the services layer and avoids sending an empty array to PostgREST.
- The method signature stays identical: `Future<void> reorderItems(List<String> ids) async` — no caller changes needed (also, grep confirmed 0 callers today).
